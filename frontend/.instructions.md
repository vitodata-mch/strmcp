# Frontend Code Standards — Vitodata Live Diktat

## Stack

- Next.js 15 (App Router), React 19, TypeScript 5 (strict)
- **Do NOT upgrade to Next.js 16** — causes crashes. Stay on 15.x.
- Tailwind CSS 4 (CSS-first config, PostCSS), shadcn/ui (base-nova style)
- socket.io-client for WebSocket, lucide-react for icons
- All UI text in **German** (Swiss orthography: ss not ß)

## Architecture Rules

### Component Structure

- **Server Components by default.** Only add `"use client"` when the component needs hooks, event handlers, or browser APIs.
- One component per file. Name file same as the default export (PascalCase for components, kebab-case for files).
- Keep components under 200 lines. Extract sub-components or hooks if longer.
- Place shared UI primitives in `components/ui/`. Place feature components in `components/`.
- Place custom hooks in `lib/`. Name them `use-*.ts`.

### File Organization

```
app/           → Pages & layouts only (App Router conventions)
components/    → Feature components + theme-related
components/ui/ → shadcn/ui primitives (do NOT modify generated code unless necessary)
lib/           → Hooks, utilities, constants, types
```

### State Management

- Use React hooks (`useState`, `useReducer`, `useRef`, `useCallback`, `useMemo`) — no external state library.
- Complex feature state belongs in a custom hook (like `useDictation`), not scattered across components.
- Use `useRef` for mutable values that should not trigger re-renders (timers, MediaStream handles, AudioContext).
- Always clean up side effects in `useEffect` return functions (close streams, clear intervals, remove listeners).

## TypeScript

- **Strict mode is ON** — never use `any`. Use `unknown` + type narrowing instead.
- Export interfaces/types from the file that defines them. Import where needed.
- Prefer `interface` for object shapes, `type` for unions/intersections/aliases.
- Use `Record<K, V>` for dictionaries, not `{ [key: string]: V }`.
- Define event handler types explicitly — avoid inline `(e: any) => ...`.
- Use `satisfies` operator for type checking without widening.

```typescript
// Good
const LABELS = {
  subjective: { short: "S", label: "Subjektiv" },
  objective:  { short: "O", label: "Objektiv" },
} as const satisfies Record<string, { short: string; label: string }>;

// Bad
const LABELS: any = { ... };
```

## Styling

### CSS Variables First

- **Never hardcode colors.** Always use CSS variables defined in `globals.css`:
  - `--background`, `--foreground`, `--surface`, `--border`, `--theme-accent`
  - `--provisional` (amber, for uncertain text), `--danger`, `--success`
- Reference them via Tailwind: `bg-[var(--surface)]`, `text-[var(--theme-accent)]`
- If adding a new semantic color, add it in ALL THREE theme blocks in `globals.css`.

### Tailwind v4 Conventions

- Use Tailwind utility classes, not inline styles (except dynamic values like `style={{ transform }}`).
- Use `cn()` from `lib/utils.ts` for conditional classes:
  ```typescript
  className={cn("base classes", condition && "conditional-class")}
  ```
- Prefer responsive utilities over media queries: `sm:`, `md:`, `lg:`.
- Use `data-[state=active]:` for stateful styles on shadcn/ui components.

### Theme Compatibility

- All new UI must look correct in all 3 themes: `obsidian`, `sardinia`, `forest`.
- Test glassmorphism effects (`backdrop-blur`) in sardinia theme specifically.
- Avoid `bg-black` or `bg-white` — use `bg-[var(--background)]` and `text-[var(--foreground)]`.

## Component Patterns

### shadcn/ui Components

- Import from `@/components/ui/...` — never import @base-ui/react directly in feature code.
- Use the `variant` and `size` props from CVA definitions. Don't override variant styles with raw Tailwind.
- To add new shadcn components: `npx shadcn@latest add <component-name>`.

### Event Handlers

```typescript
// Good — named handler, clear intent
const handleFileUpload = async (e: React.ChangeEvent<HTMLInputElement>) => {
  const file = e.target.files?.[0];
  if (!file) return;
  // ...
};

// Bad — inline complex logic
<input onChange={async (e) => { /* 20 lines */ }} />
```

### Refs Pattern for Imperative Resources

```typescript
const streamRef = useRef<MediaStream | null>(null);
const ctxRef = useRef<AudioContext | null>(null);
const timerRef = useRef<ReturnType<typeof setInterval> | null>(null);

// Always clean up
useEffect(() => {
  return () => {
    ctxRef.current?.close();
    streamRef.current?.getTracks().forEach(t => t.stop());
    if (timerRef.current) clearInterval(timerRef.current);
  };
}, []);
```

## Socket.IO / WebSocket

- Socket.IO connects to `window.location.hostname:8003` (direct, not through Next.js proxy — WS upgrades can't be proxied by rewrites).
- Always handle `connect`, `disconnect`, `connect_error` events.
- Remove event listeners in cleanup: `socket.off("event", handler)`.
- Use typed event handlers — define an interface for incoming/outgoing events.
- Audio chunks emitted as `ArrayBuffer` (PCM 16-bit signed, 16kHz mono).

## Audio Processing

- Sample rate: 16000 Hz, mono, 16-bit signed PCM.
- Use `ScriptProcessorNode` (or `AudioWorkletNode` for future upgrade) with 4096 buffer size.
- Level metering: 12 bands, RMS calculation with gain factor.
- VAD gate: only emit chunks when RMS > threshold (0.01).
- Float32 → Int16 conversion must clamp to [-1, 1] before scaling.

## Performance

- **Memoize expensive computations** with `useMemo` — especially derived values from transcript state.
- **Stable callback references** with `useCallback` — critical for Socket.IO event handlers that are added/removed.
- Avoid creating new objects/arrays in render body — causes unnecessary child re-renders.
- Level meter updates come at ~60fps from audio processing — use `requestAnimationFrame` or throttle state updates.
- Large transcript text: use `scrollIntoView({ behavior: "smooth" })`, not `scrollTop` manipulation.

## Error Handling

- Wrap async operations in try/catch. Show user-facing errors in German.
- Audio API failures (mic permission denied, no device): show actionable message.
- Socket.IO disconnects: show reconnecting state, auto-reconnect is enabled.
- File upload: validate file size (max 100 MB) and type before emitting.
- Never swallow errors silently — at minimum `console.error` for debugging.

## Accessibility

- All interactive elements must have `aria-label` or visible text label.
- Mic button: `aria-label="Aufnahme starten"` / `"Aufnahme stoppen"`.
- File input: associate with `<label>`.
- Use `aria-live="polite"` on transcript container for screen reader updates.
- Tabs: ensure proper `role="tablist"`, `role="tab"`, `role="tabpanel"` (shadcn handles this).
- Color contrast: ensure text is readable in all 3 themes (test with browser devtools).

## Security

- **No patient data in console.log** — transcription content is PHI.
- Never expose API keys client-side. Backend handles all GPUStack auth.
- Validate file types on upload (don't trust file extension alone).
- Socket.IO transport: prefer `websocket` over `polling` for data integrity.
- CSP: avoid `eval()`, `innerHTML`, or `dangerouslySetInnerHTML`.

## Don'ts

- Don't use `any` type — ever.
- Don't hardcode colors — use CSS variables.
- Don't modify shadcn/ui `components/ui/` files unless truly necessary.
- Don't add external state management (Redux, Zustand, Jotai) — React hooks suffice for this app.
- Don't use `useEffect` for derived state — compute inline or with `useMemo`.
- Don't create class components — functional only.
- Don't add dependencies without checking bundle size impact.
- Don't use `window.` globals without SSR guards (`typeof window !== "undefined"`).
- Don't log PHI (patient transcript content) in production.
